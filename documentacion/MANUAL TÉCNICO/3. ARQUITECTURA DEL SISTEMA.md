# 3. ARQUITECTURA: ENFOQUE MONOLÍTICO

La arquitectura elegida para FAMA es de tipo monolítico integrado, lo que significa que todo el código de la aplicación reside en un único proyecto y se despliega en un único servidor. Esta decisión de diseño está justificada por la naturaleza del proyecto: un sistema de uso interno con un volumen de usuarios acotado y unos requisitos de escalabilidad moderados. El enfoque monolítico simplifica el despliegue, el mantenimiento y la depuración, siendo la opción más pragmática para el contexto institucional.

## 3.1 Capa Frontend – Presentación

Es la interfaz visual con la que interactúa el usuario desde su navegador. Se ha construido con HTML5 y el motor de plantillas Jinja2, que permite generar páginas dinámicas desde el servidor de forma sencilla e integrada con Flask. El aspecto visual se basa en el framework Bootstrap como fundamento del sistema de rejilla y componentes, complementado con hojas de estilo CSS personalizadas para adaptar la identidad visual de la aplicación.

## 3.2 Capa Backend – Lógica de negocio

El backend está desarrollado en Python utilizando el microframework Flask. Esta capa es responsable de procesar las peticiones del usuario, aplicar las reglas de negocio (incluyendo el sistema de roles y permisos), interactuar con la base de datos y generar las respuestas HTML mediante las plantillas Jinja2. Flask ha sido elegido por su ligereza, flexibilidad y la amplia disponibilidad de extensiones que cubren necesidades como autenticación, gestión de sesiones y validación de formularios.

## 3.3 Capa de Base de Datos – Almacenamiento

Para el almacenamiento persistente se utiliza MongoDB, una base de datos NoSQL orientada a documentos. La conexión con Python se realiza a través de la librería PyMongo. La elección de MongoDB se justifica por la naturaleza heterogénea de los datos gestionados por FAMA: los documentos de cada módulo (vivienda, movilidad, compraventa, ocio) presentan estructuras variables que se adaptan de forma natural al modelo de documentos JSON/BSON de MongoDB, sin la rigidez de un esquema relacional fijo.
