# 8 PLIEGO DE SEGURIDAD - FAMA -

## 8.1 Objetivo

Establecer los requisitos mínimos de seguridad para el desarrollo, despliegue y operación de la aplicación web FAMA.

El objetivo es proteger la confidencialidad, integridad y disponibilidad de los datos, así como garantizar el correcto control de accesos y la trazabilidad de las acciones de los usuarios.

## 8.2 Alcance

Aplica a:
- La aplicación backend Flask.
- Las rutas de autenticación, gestión de anuncios y administración.
- La base de datos MongoDB.
- El almacenamiento de archivos subidos.
- El entorno de despliegue Docker y la configuración de producción.

No incluye hardware físico específico, pero sí el entorno Docker donde se ejecuta la aplicación.

## 8.3 Marco de seguridad

### 8.3.1 Principios generales

- Principio de menor privilegio: cada rol solo debe poder realizar lo estrictamente necesario.
- Defensa en profundidad: aplicar controles en la aplicación, en la base de datos y en el entorno de ejecución.
- Seguridad por diseño: evitar almacenar datos sensibles en texto plano.
- Trazabilidad: registrar acciones relevantes en logs.

### 8.3.2 Roles y controles de acceso

Roles definidos:
- `usuario`.
- `gestor`.
- `admin`.

Cada petición que modifica datos debe verificar explícitamente el rol del usuario.
Las operaciones de edición y eliminación de anuncios deben limitarse a:
- usuario: sus propios anuncios.
- gestor: anuncios y usuarios, pero sin eliminar cuentas de administrador.
- admin: administración completa.

## 8.4 Autenticación y gestión de credenciales

### 8.4.1 Contraseñas

- Las contraseñas deben ser validadas con criterios de fortaleza.
- Deben almacenarse solo como hashes seguros (por ejemplo, `werkzeug.security.generate_password_hash`).
- No se debe guardar ninguna contraseña en texto plano.
- El restablecimiento administrativo debe forzar cambio de contraseña en el siguiente inicio de sesión.
- Contraseña por defecto tras reseteo: `fama1234`, pero siempre con cambio obligatorio.

### 8.4.2 Registro y validación de usuarios

- El registro exige datos obligatorios y valida formato de correo electrónico.
- El usuario debe permanecer en estado `pendiente de validación` hasta que un administrador confirme la cuenta.
- Usuarios desactivados o no validados pierden la sesión activa y deben volver a autenticarse.

### 8.4.3 Sesiones

- Utilizar `SECRET_KEY` robusto y almacenado fuera del código en variables de entorno.
- El valor por defecto solo debe usarse en entornos de desarrollo.
- Las cookies de sesión deben configurarse como `HttpOnly` y, en producción, `Secure`.
- Implementar expiración de sesión razonable y limpieza de sesiones inválidas.

## 8.5 Control de entradas y validación

- Validar todos los datos recibidos desde formularios antes de procesarlos.
- Sanitizar valores de nombre, correo y campos de texto para evitar inyección y XSS.
- Validar tamaño y tipo de archivos subidos.
- Rechazar archivos cuyo tipo no esté en la lista blanca de extensiones permitidas.

## 8.6 Gestión de ficheros subidos

- Almacenar archivos en rutas controladas dentro de `static/uploads/<subcarpeta>/`.
- Utilizar `secure_filename` para evitar rutas maliciosas.
- Limitar el tamaño máximo de carga a 5 MB (`MAX_CONTENT_LENGTH = 5 * 1024 * 1024`).
- Validar extensiones permitidas: `png`, `jpg`, `jpeg`, `gif`, `webp`.
- No ejecutar ni servir archivos con extensiones no permitidas.
- Borrar archivos asociados cuando se elimina un anuncio o un recurso relevante.

## 8.7 Protección de datos

- No exponer datos sensibles en la sesión de usuario.
- Guardar únicamente los datos estrictamente necesarios en `session`.
- Proteger la información personal y evitar fugas en mensajes o logs.

## 8.8 Seguridad de la base de datos

- Conectar a MongoDB mediante URI segura y con credenciales gestionadas fuera del repositorio.
- No incluir credenciales en el código fuente o en `docker-compose.yml` sin encriptar.
- Limitar privilegios de la cuenta de aplicación en MongoDB al mínimo necesario.
- Habilitar backups periódicos de la base de datos.

## 8.9 Registros y auditoría

- Registrar acciones relevantes: creación y edición de usuarios, creación y edición de anuncios, accesos de administrador, restablecimiento de contraseñas.
- Guardar fecha, hora, usuario y tipo de acción.
- Proteger los logs contra modificaciones no autorizadas.
- Permitir la exportación controlada de logs (por ejemplo, a PDF).

## 8.10 Protección contra amenazas comunes

### 8.10.1 Inyección de código

- Usar consultas parametrizadas de MongoDB y no concatenar filtros con datos sin limpiar.
- Evitar inyecciones en operaciones sobre la base de datos y en cualquier uso de datos de entrada.

### 8.10.2 Cross-Site Scripting (XSS)

- Escapar siempre el contenido dinámico en plantillas Jinja2.
- No confiar en datos introducidos por el usuario al mostrarlos en el frontend.

### 8.10.3 Cross-Site Request Forgery (CSRF)

- Implementar tokens CSRF en formularios que modifican datos.
- Para Flask, usar `Flask-WTF` u otro mecanismo de protección CSRF.

### 8.10.4 Control de sesiones y cookies

- Establecer `SESSION_COOKIE_HTTPONLY = True`.
- En producción, establecer `SESSION_COOKIE_SECURE = True` y `SESSION_COOKIE_SAMESITE = 'Lax'` o `Strict`.

## 8.11 Despliegue seguro

- No ejecutar Flask en modo `debug` en producción.
- Asegurar que `FLASK_DEBUG` se desactive en el entorno de producción.
- Ejecutar la aplicación tras un servidor WSGI adecuado (`gunicorn` o similar) en producción.
- Configurar Docker para que el contenedor no exponga puertos innecesarios.
- Aislar MongoDB en una red privada de Docker y no exponerlo directamente a Internet.

## 8.12 Gestión de secretos y configuración

- Todas las claves secretas, URIs y credenciales deben inyectarse mediante variables de entorno.
- No versionar `SECRET_KEY`, `MONGO_URI`, contraseñas ni datos sensibles.
- Mantener un `README` de despliegue que describa los valores de entorno requeridos.

## 8.13 Recomendaciones adicionales

- Actualizar dependencias de Python periódicamente.
- Realizar revisiones de seguridad antes del despliegue final.
- Probar funciones de autenticación, gestión de sesiones y subida de archivos en un entorno de staging.
- Revisar y auditar el código de rutas protegidas con `@login_required` y control de roles.
- Considerar la implementación de límites de tasa para intentos de login.

## 8.14 Plan de contingencia

- Disponer de un procedimiento para restaurar datos desde backups.
- Tener un canal de comunicación interno para notificar incidentes de seguridad.
- Documentar los pasos para revocar sesiones y cambiar claves en caso de compromiso.

## 8.15 Cumplimiento y revisión

- Revisar el pliego de seguridad antes de cada entrega importante.
- Validar que la aplicación cumple con los requisitos de autenticación, control de accesos y gestión de archivos.
- Actualizar este documento si se introducen nuevos módulos o funciones que afecten al tratamiento de datos.
